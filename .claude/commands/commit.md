stock-agent 프로젝트에 git commit 하나를 생성한다. 아래 절차를 순서대로 따른다.

## 1. 상태 파악 (병렬 실행)

한 메시지 안에서 다음 Bash 명령들을 **병렬**로 실행한다:

- `git status` — 추적/미추적 변경 파일
- `git diff` — 스테이지되지 않은 변경
- `git diff --cached` — 스테이지된 변경
- `git log --oneline -10` — 기존 커밋 메시지 스타일 참고 (비어 있으면 새 규약을 정함)

사용자가 준 추가 힌트가 있으면 반영한다: $ARGUMENTS

## 2. 커밋 메시지 작성

- 전체 변경을 훑어 **무엇을 왜 바꿨는지**를 1~2문장으로 요약. 파일별 단순 나열은 지양.
- 언어: **한국어** (이 프로젝트의 기본 소통 언어). 고유명사·용어는 영어 그대로 가능.
- 명령형 또는 단정형, 현재시제.
- 접두어는 의미가 명확할 때만 사용: `docs:`, `feat:`, `fix:`, `refactor:`, `chore:`, `test:`.
- 제목 한 줄이 원칙. 3줄 이상이 되면 빈 줄 후 본문으로 분리.

## 3. 안전 점검

스테이지 직전 반드시 확인:

- **민감정보 파일 차단**: 경로·파일명에 `.env`, `secret`, `_key`, `credentials`, `token` 같은 키워드가 포함되면 **스테이지 중단하고 사용자에게 경고**. 확인 없이 추가 금지. KIS APP_KEY/SECRET, TELEGRAM_TOKEN 등이 평문으로 스테이지되어 있지 않은지 `git diff --cached`로 재확인.
- **`git add -A` / `git add .` 금지.** 파일명을 명시해 개별 스테이지한다.
- `.gitignore` 대상(`.env`, `data/`, `logs/`)이 섞여들지 않았는지 `git status` 재점검.

## 4. 문서 동기화 확인 (프로젝트 정책)

`CLAUDE.md`의 **문서 동기화 정책**에 따라, 아래 유형의 변경이 포함되어 있다면 `CLAUDE.md` / `README.md` / `plan.md` 중 해당 문서도 같은 커밋에 포함돼야 한다:

- Phase 진입/완료, 산출물 달성
- 리스크 한도·전략 파라미터 변경
- 기술 스택 교체
- 디렉토리 구조 추가
- 새 실행 가능 명령/스크립트 추가
- 새 결정 도입 또는 기존 결정 번복

문서 갱신이 누락된 상태로 `/commit`이 호출됐다고 판단되면, 먼저 **"문서 동기화가 필요해 보입니다 — 먼저 업데이트할까요?"** 라고 사용자에게 확인한다. 사용자가 "그냥 커밋"을 명시하면 그대로 진행.

## 5. 커밋 생성

포매팅 보존을 위해 HEREDOC을 사용한다:

```bash
git commit -m "$(cat <<'EOF'
<제목 한 줄>

<선택적 본문>

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
EOF
)"
```

## 6. 결과 확인

- 커밋 직후 `git status`로 워킹 트리 깨끗함을 확인.
- 사용자에게 **커밋 해시 + 제목 한 줄**로 간결히 보고.

## 금지 사항

- `--amend` 금지 (사용자가 명시적으로 요청하지 않는 한). pre-commit 훅 실패 시 **수정 후 새 커밋**을 만든다 — 이전 커밋을 덮어쓰지 않는다.
- `--no-verify`로 훅 건너뛰기 금지 (사용자 명시 요청 시만).
- `git push` 금지. 이 명령은 로컬 커밋까지만.
- 변경이 없으면 빈 커밋을 만들지 말고 "변경 없음"을 보고 후 종료.
